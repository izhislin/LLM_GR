## 5. Оценка масштабируемости

### Текущая пропускная способность (1 GPU, последовательная обработка)

| Метрика | Значение |
|---|---|
| Файлов в час | ~90 |
| Минут аудио в час | ~330 мин (~5.5 часов) |
| Одновременных потоков | 1 |

### Узкие места

1. **Ollama (CPU-инференс)** — 85% времени. Qwen3-8B Q4_K_M работает на CPU, GPU используется только для GigaAM.
2. **Последовательные LLM-вызовы** — 3 промпта выполняются один за другим, нет параллелизма.
3. **Таймауты** — при высокой нагрузке Ollama может не успевать за 120 сек.

### Пути масштабирования до 50 одновременных звонков

**Вариант A — Параллельные воркеры (минимальные изменения)**
- Запуск нескольких процессов pipeline параллельно.
- GigaAM: GPU-память (16GB) позволяет ~2–3 параллельные транскрибации.
- Ollama: может обслуживать несколько запросов (очередь), но при CPU-инференсе деградация линейная.
- Ожидаемый прирост: 2–3x → ~270 файлов/час.

**Вариант B — GPU-инференс для LLM (vLLM / llama.cpp с GPU offload)**
- Перенос Qwen3-8B на GPU через vLLM или llama.cpp с полным GPU offload.
- RTX 5060 Ti (16GB) достаточно для Q4-квантизованной 8B модели.
- Ожидаемое ускорение LLM: 5–10x → обработка файла за 5–10 сек.
- Проблема: GPU будет разделяться между GigaAM и LLM.

**Вариант C — Разделение GPU-нагрузки (продакшн)**
- Выделенный GPU для ASR (GigaAM) + выделенный GPU для LLM (vLLM).
- Пайплайн как очередь: ASR-воркер → LLM-воркер.
- Масштабирование: при 2 GPU — до ~500 файлов/час.
- Для 50 одновременных звонков (средняя длительность ~3 мин): нужно обрабатывать ~1000 звонков/час → потребуется 2–4 GPU.

**Вариант D — Объединение LLM-вызовов**
- Сейчас 3 отдельных запроса к LLM (суммаризация, оценка, извлечение).
- Можно объединить в 1 промпт → экономия ~60% LLM-времени.
- Риск: качество ответа может снизиться при сложных промптах.

### Рекомендация

Для тестового стенда: **Вариант D** (объединить промпты) + **Вариант B** (GPU-инференс для LLM). Это даст ~10–15x ускорение при минимальных затратах. Для продакшна с 50+ звонками: **Вариант C** с 2–4 GPU.

## 6. Проблемы и рекомендации

### Выявленные проблемы

| Проблема | Критичность | Рекомендация |
|---|---|---|
| Таймаут Ollama (1 из 18 файлов) | Средняя | Увеличить `OLLAMA_TIMEOUT` до 180 сек или добавить 3-ю попытку |
| Оценка 1/10 для разговора с ассистентом | Низкая | Добавить детекцию «нечеловеческих» разговоров (IVR, автоответчик) |
| Monkey-patch GigaAM и pyannote | Средняя | Создать форки или патч-скрипт для воспроизводимости |
| Cold start ~70 сек | Низкая | В продакшне модель будет загружена один раз при старте сервиса |

### Следующие шаги

1. Ревью качества транскрибации (ручная проверка 3–5 файлов)
2. Тюнинг промптов для оценки качества (слишком строгое приветствие?)
3. Эксперимент с объединением LLM-промптов в один
4. Тест GPU-инференса для LLM (vLLM / llama.cpp)
5. Нагрузочное тестирование с параллельными воркерами
